# 设计师与 Cursor 高效协作指南

> 写给 UX/UI 设计师：如何用更少轮对话，把想法变成可上线的作品集。  
> 基于真实项目「申文靖 UX Portfolio」的协作经验整理。

---

## 一、先建立正确预期

### 你和 Cursor 各自擅长什么

| 你（设计师） | Cursor（AI） |
|-------------|-------------|
| 视觉判断、信息层级、文案、案例叙事 | 改代码、查路径、构建、部署排查 |
| 决定「要什么感觉」 | 实现「怎么写进 CSS/数据文件」 |
| 验收最终效果 | 解释 Git/Netlify 报错含义 |

**关键认知**：Cursor 改的是**源码**，不是浏览器里已经打开的旧页面。  
源码变了 → 必须刷新 dev 页面，或重新 build 才能看到线上变化。

---

## 二、避免重复工作的 5 条铁律

### 1. 一条消息说清「目标 + 位置 + 参考」

❌ 低效（会导致多轮返工）：
> 「封面不对，改一下」  
> 「还是空白」  
> 「间距再调调」

✅ 高效：
> 「作品目录第 3 张卡片（商详改版）的封面，改用详情页 hero 里的鲜花图 + 牛奶图；默认态加 #000 30% 蒙层；悬停时显示『查看详情』，蒙层在图片上方。」

**位置怎么给**：Cursor 支持 DOM Path（浏览器检查元素复制），或说：
- 区块名：`#catalog` 第三张卡、`#pdp-redesign` hero
- 文件名：`portfolio.js`、`main.css`

**参考怎么给**：
> 「样式对齐项目 1 目录卡片的悬停蒙层」  
> 「数据从详情页二期结果区读取，不要手写第二份」

---

### 2. 区分「开发预览」和「发布预览」

很多「改了没生效」是看错页面：

| 你想做什么 | 正确打开方式 |
|-----------|-------------|
| 改 UI、改文案 | 终端运行 `npm run dev` → `http://localhost:5173` |
| 检查即将上传的包 | `npm run build` → `npm run preview` |
| 更新 Netlify | build 后，拖 **`dist` 文件夹** 到 Netlify |
| ❌ 不要 | 双击 `dist/index.html`（容易看到旧效果或空白） |

**一句话记**：日常改动用 dev；要上线前 build 一次。

---

### 3. 部署问题要带「线上证据」

❌ 「Netlify 发布后空白」  
✅ 「Netlify 空白，URL 是 `xxx.netlify.app`，F12 Network 里 JS 404」+ 截图

Cursor 可以直接对比线上 HTML 和本地 build，一次定位是路径问题还是未 rebuild。

---

### 4. Git 操作分清楚「改代码」和「写说明」

- **Summary**：Commit 的简短说明（写在 GitHub Desktop 左下角）
- **Description**：可选补充
- 改 `vite.config.js` 是在编辑器里改文件，不是在 Summary 里写文字

Push 失败（SSL 报错）通常是**网络问题**，不是代码错了；换热点/VPN 再 Push，不必让 Cursor 重改一遍代码。

---

### 5. 验收后再要 Summary

流程建议：

```
描述需求 → Cursor 改代码 → 你本地 npm run dev 确认 → 满意后再 Commit
```

避免：还没看到效果就 Commit，发现问题又要「再改一版」—— 历史 commit 变乱，自己也难回溯。

## 三、设计师专用「提问模板」

复制填空即可：

### 模板 A：改界面

```
【目标】（一句话，要达成什么视觉效果）
【位置】DOM Path 或 #区块 / 第几张卡片
【参考】（可选）复用 xxx 区域的图/样式/数据
【状态】（可选）默认态 / 悬停态 / 移动端
【验收】npm run dev 下查看
```

### 模板 B：改内容/数据

```
【目标】把 xxx 文案/指标显示到 yyy 位置
【数据源】详情页「二期结果」里的三个指标（不要重复录入）
【文件】如已知：portfolio.js 里 pdp-redesign 项目
```

### 模板 C：部署/线上问题

```
【现象】空白 / 图不显示 / 样式丢失
【URL】线上地址
【已做】是否已 npm run build、是否重新上传 dist
【截图】Network 或 Console 报错（如有）
```

## 四、本项目「改哪里」速查（减少盲目探索）

| 想改的内容 | 优先文件 |
|-----------|---------|
| 项目文案、指标、图片路径 | `src/data/portfolio.js` |
| 布局、颜色、蒙层、间距 | `src/styles/main.css` |
| 页面结构、交互逻辑 | `src/main.js` |
| 部署路径、构建方式 | `vite.config.js` |
| 换图片文件 | `public/images/` 下对应目录 |

**数据复用原则**：目录卡片指标读 `metricGroups` / `metrics`；封面可复用 `heroShowcase`，避免同一张图写两遍路径。

## 五、常见「重复劳动」及一次说清的办法

| 重复劳动 | 一次说清 |
|---------|---------|
| 改完看不到 | 「我需要在 dev 下验收，请改 src 并告诉我是否还要 build」 |
| 线上还是旧的 | 「请 rebuild dist 并说明 Netlify 要拖哪个文件夹」 |
| 蒙层挡住按钮 | 「overlay z-index 要在图片之上，默认 30% 黑，hover 50% + 文案」 |
| 间距反复微调 | 「在现有基础上再缩小/放大 20%」或给具体 px/% |
| 数据加了目录没显示 | 「metricGroups 已加，请确认 getCatalogMetrics 能读到并 rebuild」 |

## 五 B、三类高发误伤（根因 + 怎么避免）

### 1. 换一张图，却动了别的图 / 圆角

**根因（AI 侧）**
- 需求说「改封面图」，AI 可能去改整段 `heroShowcase` 数组，顺手删掉后 2 张
- 改圆角时用了过宽选择器（如所有 `img`），误伤其它设计图
- 「复用」被理解成「改数据源」而不是「展示时取前 2 张」

**你怎么说更稳**
> 「只替换 `pdp-flowers.png` 这一张路径；`heroShowcase` 仍保留 4 张，不要删后两张；不要改其它项目图片和圆角。」

**验收**：改完打开相关项目 hero，确认张数、圆角与改前一致。

### 2. Demo 不知何时不自动播放了

**根因**
- Demo 依赖悬停/进入视口触发（`bindPdpHoverAutoplay`、`is-playing`、定时器、iframe 消息）
- 改样式/结构时误删 class、挡住 hover、或 JS 绑定顺序被破坏
- 改配色/布局时「整理」了 demo 相关 CSS，副作用未测

**你怎么说更稳**
> 「这次只改背景色，不要动任何 demo 的 JS/自动播放逻辑；改完请悬停验证路径 demo / 信任 demo 仍循环。」

**验收**：每次改完项目三交互区，悬停看 demo 是否仍自动播。

### 3. 白底改了，字还是浅色看不清

**根因**
- 站点默认深色主题（浅色字）；只加白底、未改文字/边框色
- 用「白底叠在深色区块上」而不是把该区块真正改成浅色主题
- 未成套改：标题、正文、指标、分割线、卡片边框

**你怎么说更稳**
> 「项目三数据结论整块改成白底浅色主题：背景白，标题/正文/数字用深色，边框用浅灰；不要叠一层白盖在深色上。」

**验收**：白底上所有文字对比度够，无浅灰字贴白底。
